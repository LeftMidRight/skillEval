## 2026-06-01 — 模块 1 完成

### 设计方案

4 模块评测框架，总分 1~10 分制：
- 模块 1（30%）：内容还原度 — CER / TEDS+Cell F1 / 数值 F1
- 模块 2（25%）：结构组织度 — 噪声比率 / 节段覆盖 / 内容连续性
- 模块 3（25%）：鲁棒性与边界 — 异常文件 + A~G 场景
- 模块 4（20%）：业务可用性 — 下游任务准确率 + 场景衰减

### 模块 1 最终结果（25 家公司）

| 维度 | 平均分 |
|------|--------|
| 文本准确率（CER vs Mineru） | 5.9/10 |
| 表格还原度（TEDS + Cell F1） | 2.9/10 |
| 数值匹配率（F1 vs XBRL） | 3.9/10 |
| **模块 1 总分** | **4.1/10** |

### 实现中遇到的问题与解决

| # | 问题 | 解决方案 |
|---|------|---------|
| 1 | LAS 输出是 HTML `<table>` 而非 markdown 表格 | 新增 `utils.py` 中的 `_TableHTMLParser` + `TableTree` 类 |
| 2 | 跨页表格被 LAS 拆成多个 `<table>` 片段 | `extract_all_tables()` 中实现按 section heading 合并逻辑 |
| 3 | XBRL 是精简版（79 行），LAS 是完整版（178 行），FP 天然高 | FP 策略改为"只罚 XBRL 缺的，不罚 LAS 多的" |
| 4 | 数值匹配 FP 太高（819 vs 180） | 改用表格内提取替代全文正则提取，FP 从 642→268，F1 从 0.35→0.57 |
| 5 | CER 打分阈值太紧（0.005 才算 10 分） | 财报场景特调：0.03→10 分, 0.10→8 分, 0.22→6 分 |
| 6 | 某些公司在 test.txt 中找不到 XBRL（实际在 dev.txt 中） | `Evaluator` 支持 `xbrl_paths` 合并多数据集 |
| 7 | Windows GBK 编码报 UnicodeEncodeError | 测试中用 ASCII 替代 unicode 字符 |
| 8 | `table_composite_score` 错误地把原始 TEDS/CellF1 值直接加权 | 改为先各自转 1~10 分再加权：`0.6×TEDS分 + 0.4×CellF1分` |
| 9 | text CER 用 `median_cer` 但值为 NaN，`get()` 不 fallback | 改用 `or` 链 + NaN 显式检测 |

### 当前已知局限

1. **现金流量表匹配失败率高** — 原因：XBRL 是"合并现金流量表"，LAS 输出有时是"母公司现金流量表"，标题不匹配
2. **XBRL 是精简版，LAS 是完整版** — Cell F1 无法衡量 LAS 多提取的内容质量
3. **CER 缺少绝对 ground truth** — Mineru 是最优参照但不完美
4. **HTML colspan 被 LAS 丢失** — TEDS 能检测到但无法修复（需 LAS 侧改进）
5. **LAS 不输出元素类型标签** — `text_blocks` 全部为 `"label": "text"`，无法评测元素识别能力

## 2026-06-01 — 模块 2 完成

### 实现方案

模块 2 三个子指标（均无需外部参照，纯规则驱动）：

| 子指标 | 检测方法 | 得分映射 |
|--------|---------|---------|
| 2.1 噪声比率 | 正则检测页码 + Counter 检测重复行(≥3x) + 关键词检测签名栏/审计标记 | noise_chars/total_chars → 分档映射 |
| 2.2 节段覆盖 | Markdown 标题行 + 纯文本表名行双通道检测 | 3 核心必须全有=8 分，每缺一个-3 分，审计+2 分 |
| 2.3 内容连续性 | 噪声行切分→纯净块长度分布 | avg_block/300 字符=分(上限 10) |

### 模块 2 最终结果（25 家公司）

| 维度 | 平均分 |
|------|--------|
| 噪声比率 | 7.7/10 |
| 节段覆盖 | 8.1/10 |
| 内容连续性 | 4.4/10 |
| **模块 2 总分** | **6.8/10** |

得分分布：5分×1, 6分×4, 7分×18, 8分×2

### 模块 2 发现

1. **LAS 三大核心报表全部保留** — 25 家公司资产负债表/利润表/现金流量表均被检测到
2. **审计报告**则因 LAS 不提取纯文本中的审计报告标题（无 `#` 标记），大多数未能检测
3. **连续性差是最大短板** — 每页 PDF 的页码 + 页眉 + 签名栏打断了正文流，噪声打断 20-30 次/份
4. **噪声本身不多**（~1.4% chars），但分布零散（每 2-3 页一次），碎片化效应显著

### 与前版方案的变化

原方案"阅读顺序正确率"和"标题层级标记率"因缺少 ground truth + LAS 不输出元素类型，改为三个能源代码检测的结构指标。

### 实现中遇到的问题与解决

| # | 问题 | 解决方案 |
|---|------|---------|
| 1 | 节段检测大量误报——LAS 输出中大量表名是纯文本行而非 `#` 标题 | 新增纯文本关键词检测通道，过滤 HTML 标签行和 `|` 表格行 |
| 2 | "现金流"关键词匹配到表格行"投资活动产生的现金流量净额" | 移除"现金流"关键词，只用"现金流量表" |
| 3 | 连续性评分 HTML 表格行被错误跳过 | 移除 `startswith("<")` 的跳过逻辑，HTML 行也是内容 |
| 4 | 噪声打断每行计一次导致 break 数虚高 | 改连续噪声行合并为 1 次打断（in_noise 状态机） |

## 2026-06-01 — 模块 2 重定义

### 问题发现

用户在审核模块 2 的原方案（噪声比率/节段覆盖/内容连续性）时提出关键质疑：

> "PDF 解析 skill 应该把页眉页脚解析出来，为什么要设为噪声？"

这个质疑揭示了原方案的根本缺陷：把"页面装饰被正确提取"（LAS 的正确行为）和"物理→逻辑转换未完成"混在了一起。

### 模块 2 重新定义

**核心定义**：PDF 物理页面结构 → 文档逻辑结构的转换质量。

三个层级：

| 层级 | 子指标 | 测什么 |
|------|--------|--------|
| 页级 | 页面装饰密度 | 物理页面残余（页码、重复页眉、签名栏）的占比 |
| 表级 | 跨页表格完整性 | 跨页表格是否被拼回连续整体（中间不应夹页码/页眉） |
| 节级 | 关键节段覆盖 | 资产负债表/利润表/现金流量表是否齐全 |

与原方案的区别：

| 旧 | 新 | 变化原因 |
|----|-----|---------|
| 噪声比率 | 页面装饰密度 | 改名——"噪声"暗示错误，"装饰密度"描述客观事实 |
| 内容连续性 | 跨页表格完整性 | 改指标——"打断"只是表象，根本问题是"表有没有被拼回去" |
| 节段覆盖 | 关键节段覆盖 | 保留，范围收窄为必备三张表 |

### 重要澄清

页面装饰密度**不是在评测 LAS 是否"做错了"**。提取页码和页眉是 LAS 的正确行为（说明没漏东西）。它量化的是**物理→逻辑转换中残留的页面装饰程度**——这是 LAS 下一步可以改进的方向。

### 待实现

- 2.2 跨页表格完整性（检测 `</table>` ... `<table>` 之间的页面标记）
- 完善后重新跑 25 家批量结果

## 2026-06-02 — 模块 1 场景拆解完成

### 实现方式

对 25 家公司的模块 1 结果按 A~G 六个场景标签分组，计算每个场景的指标均值。

### 核心发现

| 场景 | n | 总分 | 发现 |
|------|---|------|------|
| A/B/C/G | 25 | 4.1 | 这四个场景覆盖全部 25 家公司（标签有重叠），均值 = 全量基线 |
| D_多页 | 5 | **3.9** | 长文档（≥14 页）TEDS 从 0.472 降至 0.431，总分 -0.21 |
| F_多栏 | 3 | 4.1 | 与基线持平，样本小（n=3）但未观察到显著劣化 |

### 重要结论

**场景拆解按公司分组只能区分 D 和 F 这两个子集**。A/B/C/G 因为每份 PDF 同时属于多个场景（如一份 PDF 既有跨页表又有密集数值），按公司分组无法区分。

要真正区分 A/B/C/G 场景的 LAS 表现差异，需要**场景专属指标**（非通用的模块 1 指标）：
- A（跨页表格）→ 表内部片段数、跨页断裂点密度
- B（密集数值）→ 千分位逗号保留率、单位识别率
- C（页眉页脚）→ 页面装饰密度（已纳入模块 2）
- G（无边框表格）→ 无边框表的召回率（需人工标注哪些表是无边框的）

这些专属指标更适合放在**模块 3（鲁棒性与边界→A~G 场景专项）**中实现，而非简单地按公司分组重算模块 1 的通用指标。

- `scripts/scene_breakdown.py` — 场景拆解脚本
- 输出：`output/las_results/module1_scene_breakdown.json`
